## 1. 系统概述

Multi-Agent Orchestrator 是一个让 Claude Code "大哥"指挥多个 Claude Code "小弟"协同完成任务的系统。

### 1.1 核心目标

- **效率提升**: 并行执行多个子任务，大幅缩短完成时间
- **能力增强**: 每个小弟拥有完整能力，可调用所有 Skills/Agents/MCP
- **智能协作**: 小弟可以指挥其他空闲小弟，形成任务树
- **可视化管理**: Telegram 群聊实时展示任务进度和 Agent 状态
- **容错能力**: 自动错误恢复和任务重分配

### 1.2 技术栈

- **运行时**: Node.js 20+
- **语言**: TypeScript 5+
- **AI API**: Claude API (Anthropic)
- **消息队列**: 内存队列 + SQLite
- **数据库**: SQLite
- **Telegram Bot**: node-telegram-bot-api
- **测试**: Vitest

## 2. 系统架构

### 2.1 分层架构

```

│                      Telegram Bot                            │
│              (可视化交互界面 + 消息路由)                      │

                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                   Orchestrator (大哥)                        │

└─────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  Worker #1   │      │  Worker #2   │ ...  │  Worker #9   │

└──────────────┘      └──────────────┘      └──────────────┘
```

### 2.2 核心组件

#### 2.2.1 Orchestrator（大哥）

**职责**:
- 接收用户需求并理解意图
- 将复杂任务分解为可并行的子任务
- 分配任务给空闲的 Worker
- 监控所有 Worker 的执行进度
- 质量把控和结果整合
- 错误处理和任务重分配

**能力**:
- 完整的 Claude Code 能力
- 访问所有 Skills/Agents/MCP
- 任务分解和规划能力
- 全局视图和决策权

#### 2.2.2 Worker（小弟）

**职责**:
- 执行分配的子任务
- 定期报告进度（心跳机制）
- 遇到问题时请求帮助
- 可以指挥其他空闲 Worker（但不能指挥大哥）

**能力**:
- 完整的 Claude Code 能力
- 访问所有 Skills/Agents/MCP
- 可以创建子任务并分配给其他 Worker

**数量**:
- 初始: 9 个固定 Worker
- 动态: 根据需要创建临时 Worker
- 最大: 无限制（但考虑 API 限流）

#### 2.2.3 Task Manager（任务管理器）

**职责**:
- 任务队列管理（FIFO + 优先级）
- 任务状态追踪（pending/running/completed/failed）
- 任务依赖关系管理
- 任务树结构维护

**数据结构**:
- 优先级队列（heap）
- 任务树（parent-child 关系）
- 依赖关系图（DAG）

#### 2.2.4 State Manager（状态管理器）

**职责**:
- Agent 状态追踪（idle/busy/error）
- 并发控制和锁机制
- 状态持久化（内存 + SQLite）

**数据结构**:
- Agent 状态表
- 心跳记录表
- 统计信息表

#### 2.2.5 Memory Service（记忆服务）

**职责**:
- 中央化的配置和记忆管理
- 读写锁机制（多读单写）
- 缓存机制（减少文件 I/O）
- 变更通知（发布-订阅模式）

**实现**:
- 文件系统监听
- LRU 缓存
- EventEmitter 事件通知

## 3. 数据流设计

### 3.1 任务创建流程

```
用户 → Telegram Bot → Orchestrator
  ↓
任务分解
  ↓
创建子任务 → Task Queue
  ↓
分配给 Workers
```

### 3.2 任务执行流程

```
Worker 接收任务
  ↓
更新状态为 busy
  ↓
执行任务（调用 Claude API）
  ↓
定期发送心跳（每 5 分钟）
  ↓
完成任务 → 更新状态为 idle
  ↓
报告结果给 Orchestrator
```

### 3.3 错误恢复流程

```
Worker 执行失败
  ↓
报告错误给 Orchestrator
  ↓
Orchestrator 分析错误
  ↓
决策: 重试 / 重新分配 / 创建新 Worker
  ↓
执行恢复策略
```

## 4. 关键设计决策

### 4.1 为什么使用 SQLite？

- **轻量级**: 无需额外的数据库服务
- **可靠性**: ACID 事务保证
- **性能**: 本地文件访问，低延迟
- **简单**: 易于部署和维护

### 4.2 为什么使用内存队列 + SQLite？

- **性能**: 内存队列提供快速访问
- **持久化**: SQLite 提供数据持久化
- **恢复**: 系统重启后可以恢复任务队列

### 4.3 为什么使用任务树结构？

- **层次化**: 清晰的任务层次关系
- **可追溯**: 可以追溯任务的来源
- **防止循环**: 通过深度限制防止无限递归

### 4.4 为什么使用心跳机制？

- **健康检查**: 及时发现 Worker 故障
- **超时检测**: 检测长时间无响应的任务
- **负载均衡**: 了解 Worker 的实时状态

## 5. 安全考虑

### 5.1 API 密钥管理

- 使用环境变量存储 API 密钥
- 不在代码中硬编码密钥
- 使用 .env 文件（不提交到 Git）

### 5.2 并发控制

- 使用读写锁防止数据竞争
- 使用乐观锁处理并发更新
- 使用事务保证数据一致性

### 5.3 Telegram Bot 安全

- 验证用户身份
- 限制可执行的命令
- 记录所有操作日志

## 6. 性能优化

### 6.1 缓存策略

- LRU 缓存减少文件 I/O
- 缓存命中率目标 > 90%

### 6.2 并发优化

- 任务并行执行
- Worker 池动态扩展
- 负载均衡算法

### 6.3 数据库优化

- 添加索引提升查询性能
- 使用批量操作减少事务次数
- 定期清理历史数据

## 8. 部署架构

### 8.1 部署环境

- **平台**: WSL2 (Ubuntu)
- **运行时**: Node.js 20+
- **进程管理**: systemd
- **日志**: Winston + 文件日志

### 8.2 目录结构

```
/opt/multi-agent-orchestrator/
├── dist/                 # 编译后的代码
├── node_modules/         # 依赖
├── logs/                 # 日志文件
├── data/                 # SQLite 数据库
└── .env                  # 环境变量
```

### 8.3 监控和日志

- 日志级别: debug, info, warn, error
- 日志轮转: 每天一个文件，保留 30 天
- 监控指标: Token 使用、任务完成时间、错误率
