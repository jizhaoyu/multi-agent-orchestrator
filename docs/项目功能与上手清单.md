# 项目功能与上手清单

这份文档把当前项目已经实现的能力，以及一条适合第一次接手时的上手路径整理在一起，方便快速建立整体认识。

## 一、当前已经实现的功能

### 1. 核心编排能力

- 多 Agent 编排器：支持 `Orchestrator` 接收任务、分解任务、按优先级分配任务、批量调度、空闲 Worker 选择、超时监控、错误重分配、结果整合。
- Worker 执行模型：支持接收任务、执行任务、上报进度、心跳保活、请求帮助、子任务委派、销毁清理。
- 任务与状态持久化：`TaskManager` 和 `StateManager` 都走 SQLite，支持任务树、依赖检查、可执行任务筛选、运行中任务查询、Agent 统计与心跳超时检测。
- 记忆服务：`MemoryService` 支持读取、写入、缓存、watch、订阅变更、按优先级读取第一份指令文件。

### 2. 本地工作区执行

- `WorkspaceExecutor` 支持 5 类动作：读文件、找文件、写文件、跑命令、结束任务。
- 支持模糊找文件、目录树扫描、Git 状态注入、进度事件回传。
- 命令执行有安全限制，默认只允许安全白名单命令。
- 任务结束时不会直接信任模型的“完成”，而是自动进入 verifier 判定。

### 3. Codex Harness 能力

- 已实现 Generator-Verifier-Reviser 闭环。
- 已实现 `VerificationPolicy`、`VerificationCommand`、`VerificationResult`、`FailureClass`、`PermissionProfile`、`TraceEvent` 等通用类型。
- 已实现 `VerificationEngine`：自动跑 `lint`、`typecheck`、`test`、`build`、`custom` 等检查，并在失败时生成 revision prompt。
- 已实现 `TraceRecorder` 和 middleware hooks：覆盖 `before_prompt`、`after_response`、`before_tool`、`after_tool`、`before_verify`、`after_verify`、`task_failed`、`task_completed`。
- 已实现 `FailureMemoryStore`：把失败模式沉淀到 `docs/failure-catalog.md`。
- `Orchestrator.reviewResults` 已能识别 verifier verdict，而不只是看结果长度。

### 4. LLM 与 Telegram 集成

- 已支持两类模型接入：Codex/OpenAI-compatible 和 Claude。
- 已有环境变量工厂 `createAPIClientFromEnv()`。
- Telegram Bot 集成已支持命令：
  - `/task`
  - `/project`
  - `/projects`
  - `/pwd`
  - `/queue`
  - `/logs`
  - `/cancel`
  - `/status`
  - `/workers`
  - `/reset`
  - `/help`
- Telegram 侧已支持项目目录切换、按项目名搜索工作区、任务接收确认、流程阶段更新、长结果分段发送、队列快照、最近日志、软取消、Worker 状态查看。
- 已实现网络错误重试与 polling 冲突处理。

### 5. 工程化与未来项目复用

- 已有 repo 级验证链路：`lint`、`typecheck`、`test`、`build`、`verify`。
- 已有 Harness 启动脚本：`harness:init`、`harness:doctor`、`benchmarks`。
- 已提供未来项目 starter：`templates/codex-harness/`，里面有 `AGENTS.md`、`agent-map`、`verification runbook`、`failure-catalog`、`skills`、`benchmark`、`codex-harness.config.json`。
- 已有 benchmark 样例和结果样例。

## 二、当前还不算完全展开的部分

- benchmark 现在是“结果汇总器 + 样例数据”，还不是自动执行真实 agent 任务的完整评测平台。
- failure catalog 已接入，但目前还是 Markdown 目录，不是结构化数据库查询系统。
- 权限模型已经落地到命令白名单，但网络分级和更细粒度的工具权限还可以继续扩展。

## 三、建议的上手操作清单

### 第 1 步：确认工程能跑通

```bash
npm install
npm run verify
```

目标：

- 确认依赖完整
- 确认 lint、typecheck、test、build 全链路正常

### 第 2 步：检查 Harness 文档和配置是否一致

```bash
npm run harness:doctor
```

目标：

- 确认 `AGENTS.md`
- 确认 `docs/agent-map.md`
- 确认 `docs/failure-catalog.md`
- 确认 `docs/runbooks/verification.md`
- 确认 `codex-harness.config.json`

### 第 3 步：先看 benchmark 汇总入口

```bash
npm run benchmarks
```

目标：

- 先理解项目是怎么做评估入口的
- 理解 benchmark case 和 benchmark result 的关系

### 第 4 步：看 starter 模板，理解未来项目怎么复用

建议直接打开这些文件：

- `templates/codex-harness/AGENTS.md`
- `templates/codex-harness/codex-harness.config.json`
- `templates/codex-harness/docs/runbooks/verification.md`
- `templates/codex-harness/docs/failure-catalog.md`

目标：

- 理解未来项目初始化时最少要有哪些文件
- 理解 verifier 默认会依赖什么命令

### 第 5 步：看总入口和核心执行路径

建议阅读顺序：

1. `src/index.ts`
2. `src/core/orchestrator.ts`
3. `src/core/worker.ts`
4. `src/core/workspace-executor.ts`
5. `src/harness/verification-engine.ts`

目标：

- 搞清楚导出面
- 搞清楚编排层
- 搞清楚执行层
- 搞清楚 verifier 是如何卡住“误报完成”的

### 第 6 步：看测试，理解行为边界

建议优先看：

- `tests/unit/core/workspace-executor.test.ts`
- `tests/unit/harness/middleware.test.ts`
- `tests/unit/core/orchestrator.test.ts`
- `tests/unit/core/worker.test.ts`
- `tests/unit/integrations/telegram/bot.test.ts`

目标：

- 理解闭环行为是如何被测试出来的
- 理解 Telegram 命令入口已经覆盖到什么程度

### 第 7 步：跑基础示例

```bash
npm run example:basic
```

目标：

- 感受 orchestrator 的基础使用方式

### 第 8 步：如果你有 Telegram 配置，再跑 Telegram 示例

先查看：

- `.env.example`

然后执行：

```bash
npm run example:telegram
```

目标：

- 体验 Telegram 作为控制面时的实际使用方式

## 四、你阅读时最值得盯住的 4 个点

- `finish` 不是完成，只有 verifier 通过才算完成。
- 命令不是任意执行，必须落在白名单。
- 失败不是丢掉，而是会进入 failure catalog。
- future project 的起点不是 prompt，而是 starter 模板和 `codex-harness.config.json`。
